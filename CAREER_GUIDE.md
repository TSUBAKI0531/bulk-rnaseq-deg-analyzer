### パターン A: バイオインフォ専任職向け（製薬・CRO・受託解析）

**プロジェクト名:** RNA-seq 差次発現解析パイプラインの開発（個人開発）

**期間:** 2026年（約1ヶ月）

**概要:**
RNA-seq カウントデータから差次発現遺伝子（DEG）の統計検定、多角的な可視化、パスウェイエンリッチメント解析までを Python で一気通貫で実行する Web アプリケーションを開発。R 依存を排し、Python エコシステムのみで RNA-seq 解析を完結させた。

**担当範囲:** 設計・実装・デプロイの全工程を単独で担当

**技術詳細:**
- pyDESeq2（負の二項分布モデル）による DEG 検定パイプラインを構築し、低発現遺伝子フィルタリング、正規化、Wald 検定を自動化
- ボルカノプロット、MA プロット、階層クラスタリングヒートマップ（Z-score 正規化）、PCA による4種の標準可視化を自動出力
- gseapy（Enrichr API）を統合し、GO Biological Process / KEGG / Reactome 等 6 ライブラリに対する Over-Representation Analysis を Up/Down 方向別に実行
- Streamlit による GUI 化でパラメータ調整をインタラクティブに操作可能とし、非プログラマーでも利用可能な設計
- GitHub でのソースコード管理、Streamlit Cloud へのデプロイ、README・ドキュメント整備まで一貫して対応

**使用技術:** Python, Streamlit, pyDESeq2, gseapy, seaborn, matplotlib, scikit-learn, pandas, Git/GitHub

**成果物:** GitHub リポジトリ + Streamlit Cloud デプロイ済み Web アプリ
- GitHub: https://github.com/your-username/rnaseq-deg-analyzer
- App: https://your-app-url.streamlit.app

## 2. 面接での想定 Q&A

### Q1:「このプロジェクトを簡単に説明してください」

> RNA-seq のカウントデータを入力すると、差次発現遺伝子の統計検定から、ボルカノプロットやヒートマップなどの可視化、さらに GO エンリッチメント解析までを全自動で実行する Web アプリを Python で作りました。通常この解析は R の DESeq2 で行うのが標準ですが、pyDESeq2 という Python 実装を使うことで、R を使わずに Python だけで完結させています。Streamlit で GUI 化しているので、コードを書かずにブラウザ上で操作できます。

### Q2:「なぜ R ではなく Python で作ったのですか？」

> 3つ理由があります。1つ目は、Python であれば機械学習やディープラーニングのパイプラインとシームレスに統合できる点です。将来的にDEGの結果を機械学習モデルの特徴量にしたり、創薬ターゲット予測に繋げることを見据えています。2つ目は、Streamlit による Web アプリ化が容易で、ウェットラボの研究者に対してブラウザ経由でツールを提供できる点です。3つ目は、自分自身が抗体医薬品開発のウェットラボ出身なので、Python でバイオインフォと実験データ解析を一つの言語で統一できることが実務上のメリットとして大きいと考えたためです。

### Q3:「pyDESeq2 とは何ですか？DESeq2 との違いは？」

> DESeq2 は R 言語で実装された RNA-seq の差次発現解析の標準ツールで、負の二項分布モデルを使って遺伝子ごとの発現変動を統計検定します。pyDESeq2 は Owkin 社がこれを Python で再実装したもので、統計モデルは DESeq2 と同等です。R 版との主な違いは、pandas の DataFrame でデータを扱える点と、Python のエコシステム（scikit-learn, matplotlib 等）とそのまま連携できる点です。

### Q4:「GO エンリッチメント解析の結果をどう解釈しますか？」

> GO エンリッチメント解析では、DEG リストに特定の生物学的機能や経路に関わる遺伝子が統計的に多く含まれているかを検定します。例えば、Up-regulated な DEG に「immune response」や「cytokine signaling」が濃縮されていれば、処理群で免疫関連のシグナルが活性化していると解釈できます。このツールでは Up と Down を分けて解析できるので、活性化と抑制の両方向から生物学的な意味を読み取ることができます。さらに GO だけでなく KEGG や Reactome も同時に見ることで、パスウェイレベルでの全体像を把握します。

### Q5:「ボルカノプロットと MA プロットの違いは？なぜ両方出すのですか？」

> ボルカノプロットは X 軸が log2 Fold Change、Y 軸が統計的有意性（-log10 padj）で、「どの遺伝子が有意かつ大きく変動したか」を一目で把握できます。一方 MA プロットは X 軸が平均発現量（baseMean）、Y 軸が log2 Fold Change で、「発現量の低い遺伝子で偽陽性が出ていないか」を確認できます。低発現遺伝子は統計的にノイズが大きいため、ボルカノだけでは見落とす問題を MA で補完します。論文でもこの2つを並べて掲載するのが標準的なので、両方出力する設計にしました。

### Q6:「PCA で何を見ていますか？」

> 全遺伝子の発現パターンを使って、サンプル間の全体的な類似性を2次元に圧縮して可視化しています。期待通りの結果であれば、Control 群と Treatment 群が PC1 軸で分離します。もしあるサンプルだけ離れていればバッチ効果や実験のアウトライアーの可能性があるので、下流の解析前に品質チェック（QC）として使います。

### Q7:「苦労した点は？」

> pyDESeq2 が要求する入力データの形式（サンプル×遺伝子の転置行列）と、一般的な RNA-seq カウントマトリクス（遺伝子×サンプル）の向きの違いに気づくまでに時間がかかりました。また、デモデータを生成する際、負の二項分布のパラメータ設定で、実データに近い分散とカウント分布を再現するのに試行錯誤しました。さらに、GO エンリッチメント解析ではデモデータ（Gene_0001 形式）で Enrichr API が当然動かないため、デモ時にはモック結果を返し、実データ時のみ API を呼ぶ分岐設計にした点も工夫しました。

### Q8:「今後の改善予定は？」

> 3つ考えています。1つ目は GSEA（Gene Set Enrichment Analysis）のランクベース解析への対応です。現在の ORA は DEG リストの有無で二値的に判定しますが、GSEA は全遺伝子のランキングを使うので、閾値設定に依存しない解析が可能になります。2つ目は多群比較への拡張で、3群以上の条件で DEG を検出できるようにします。3つ目は Plotly によるインタラクティブ図への切替で、マウスオーバーで遺伝子情報を表示する機能を追加します。
